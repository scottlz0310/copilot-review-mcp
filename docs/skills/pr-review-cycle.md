# pr-review-cycle Skill

[日本語](pr-review-cycle.ja.md)

A skill that uses the copilot-review-mcp watch tools to wait for Copilot review completion via **async watch**, then autonomously runs the PR review response cycle.

## Overall Flow

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6
                ↑                                              ↓ WAIT / REQUEST_REREVIEW
                └──────────────────────────────────────────────┘
                                        ↓ READY_TO_MERGE
                              Phase 6.5 → Phase 6.6 → Phase 7 → Phase 8
                                        ↓ ESCALATE
                                    End (report to user)
```

---

## Phase 0: Snapshot Check

1. Call `{CRM}:get_copilot_review_status` to immediately check the current state on GitHub.
2. `status = COMPLETED` or `BLOCKED`: → Go to Phase 2 (no watch needed).
3. `status = NOT_REQUESTED`: Request a review with `{CRM}:request_copilot_review` → Go to Phase 1.
4. `status = PENDING` / `IN_PROGRESS`: → Go to Phase 1.

## Phase 1: Start Async Watch + Wait for Completion

**Record the wait start time.** When looping back from Phase 6, record a new start time (the timeout resets on each entry to Phase 1).

### 1-A: Start Watch

1. Call `{CRM}:start_copilot_review_watch` (returns immediately).
   - If an active watch for the same PR already exists, it is reused.
   - Record the `resource_uri` (`copilot-review://watch/{watch_id}`) from the response.

### 1-B: Wait for Completion

**With notification support (recommended):**
- Subscribe to `resource_uri` as an MCP resource and wait for a `notifications/resources/updated` notification.
- After receiving the notification: confirm the final status with `{CRM}:get_copilot_review_watch_status`.

**Polling fallback:**
- Call `{CRM}:get_copilot_review_watch_status` after `next_poll_seconds` seconds from the response (minimum 1 second; server default is 90 seconds).
- Follow `recommended_next_action`:

| recommended_next_action | Action |
|------------------------|--------|
| `READ_REVIEW_THREADS` | → Go to Phase 2 |
| `POLL_AFTER` | Re-poll after the specified number of seconds |
| `CHECK_FAILURE` | Report the error to the user and abort the cycle |
| `REAUTH_AND_START_NEW_WATCH` | Prompt user to re-authenticate and exit |
| `START_NEW_WATCH` | Restart the watch (→ 1-A) |

### 1-C: Timeout Handling (elapsed ≥ 15 minutes)

1. Cancel the watch with `{CRM}:cancel_copilot_review_watch`.
2. Post via `{GH}:add_issue_comment`:
   `The Copilot review completion wait timed out (15 minutes). Please resume manually.`
3. Guide the user on how to resume manually and exit.

## Phase 2: Fetch Threads

- Fetch all comment threads with `{CRM}:get_review_threads`.
- If 0 threads: Copilot may have judged no issues → Go to Phase 7.

## Phase 3: Classify & Accept/Reject (Autonomous)

Classify each comment using the criteria below and autonomously decide `accept` / `reject`:

| Category | Criteria |
|----------|----------|
| `blocking` | Runtime errors, data integrity violations, security risks, breaking changes |
| `non-blocking` | Recommended but not required: adding tests, improving logs, etc. |
| `suggestion` | Design, naming, or abstraction improvements |

For `reject`, clearly state the reason (out of scope, tracked in a separate PR, conflicts with design policy, etc.).

Present the results to the user in the table below before proceeding to Phase 4 (no approval wait):

```
| # | Thread ID | Category | Decision | Summary | Reason (reject only) |
|---|-----------|----------|----------|---------|----------------------|
```

Also determine which `fix_type` the changes fall under (used in Phase 6):

| fix_type | Applicable Cases |
|----------|-----------------|
| `logic` | Implementation logic or test-only changes |
| `spec_change` | Spec, API, or interface changes |
| `trivial` | Typos, comments, or formatting only |
| `none` | No changes (all rejected) |

## Phase 4: Fix + Commit

1. Fix only the `accept`-ed items.
2. **Granularity**: 1 thread = 1 logical change unit (atomic). If changes ripple into other threads, re-evaluate the accept/reject table before proceeding.
3. After all fixes, re-run the build and tests. Retry on failure. If unresolvable, abort the cycle and report to the user.
4. After Phase 4 is complete, make **one commit** covering all fixes (Conventional Commits format).

## Phase 5: Reply + Resolve Threads

**Reply to and resolve every thread without exception** (missing any is a potential merge blocker):

| Situation | Tool |
|-----------|------|
| Fixed thread | `{CRM}:reply_and_resolve_review_thread` |
| Rejected / no action | `{CRM}:reply_and_resolve_review_thread` |

Rejected replies must always state the reason (out of scope, tracked in a separate Issue/PR, conflicts with design policy, etc.).

## Phase 6: Cycle Evaluation

Call `{CRM}:get_pr_review_cycle_status` with the following arguments:

```json
{
  "owner": "<owner>",
  "repo": "<repo>",
  "pr": 42,
  "cycles_done": 0,
  "max_cycles": 0,
  "fix_type": "<fix_type determined in Phase 3>"
}
```

> `pr` is the PR number (integer). `max_cycles: 0` uses the server-side default (env var `MAX_REVIEW_CYCLES`, defaults to 3 if unset).
> `cycles_done` is a 0-based integer. Increment it on each entry to Phase 1 (first call: `0`, second cycle: `1`, …).

Follow `recommended_action`:

| recommended_action | Next Action |
|-------------------|-------------|
| `WAIT` | Return to Phase 1 (increment `cycles_done`) |
| `REPLY_RESOLVE` | Return to Phase 2 (unresolved threads remain) |
| `REQUEST_REREVIEW` | `{CRM}:request_copilot_review` → Return to Phase 1 (increment `cycles_done`) |
| `READY_TO_MERGE` | → Phase 6.5 |
| `ESCALATE` | Report current status to the user and end the cycle (defer to human judgment) |

## Phase 6.5: CI Check

1. Check CI status with `gh pr checks <PR number>`.
2. All jobs SUCCESS → Phase 6.6.
3. Failing jobs: Fetch logs with `gh run view <run-id> --log-failed` and analyze the cause.
   - **Fixable** (code issue, clear failure cause): Add to the accept/reject table and return to Phase 4.
   - **Hard to fix** (environment factors, flaky tests, unknown cause): Report to the user and await instructions.

## Phase 6.6: Coverage Check

1. Check for Codecov or similar coverage PR comments (skip if none → Phase 7).
2. Evaluate patch coverage, overall coverage changes, and uncovered new logic.
3. Decision:
   - **Fixable by adding tests**: Return to Phase 4 (`fix_type = logic`).
   - **Requires design changes / hard to mock**: Report to the user and await a decision.
   - **No issues**: → Phase 7.

## Phase 7: Post Summary Comment

Post the following to the PR via `{GH}:add_issue_comment`:

```markdown
## Review Response Summary

### Changes Made
- (bulleted overview)

### Accept/Reject Decisions
- accept: N items
- reject: M items
  - #<thread>: (reason)

### Unresolved Items
- (if any, with reasons)
```

## Phase 8: Merge Decision

**Do not merge autonomously.** Wait for explicit instructions from the user.

Merge conditions (must be satisfied when the user instructs a merge):
- All CI jobs SUCCESS
- 0 `blocking` comments
- 0 unresolved review threads
- All comments replied to

If any condition is not met, report the missing items to the user and await instructions.

## Notes

- Polling interval: follow `next_poll_seconds` from the response (minimum 1 second; server default 90 seconds)
- Timeout: resets to 15 minutes on each entry to Phase 1
- Re-review limit: server-side `MAX_REVIEW_CYCLES` (default 3). After `ESCALATE`, defer to human judgment
- Fix granularity: atomic per thread (1 thread = 1 logical change unit)
- Commit strategy: one commit after Phase 4 completes (Conventional Commits format)
- Phase 3 accept/reject decisions are autonomous but the result table must always be presented (for auditability)
- If test failures cannot be resolved in Phase 4, abort the cycle
- Phase 8 requires explicit user instruction (operational safety)

## Tool Reference

| Tool | Purpose |
|------|---------|
| `{CRM}:get_copilot_review_status` | Instant check of Copilot review state on GitHub |
| `{CRM}:request_copilot_review` | Request a Copilot review |

| `{CRM}:wait_for_copilot_review` | ※ Legacy blocking wait — use watch instead |

## MCP Resource Subscription (Reference)

Subscribing to the `resource_uri` returned by `start_copilot_review_watch` lets you receive `resources/updated` notifications whenever the watch state changes.

```
URI: copilot-review://watch/{watch_id}
MIME: application/json
```

After receiving a notification, read the latest state with `{CRM}:get_copilot_review_watch_status`.
